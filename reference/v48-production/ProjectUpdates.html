Change 1: Spreadsheet write locking and grouped row writes
Date implemented: 2026-04-17
Files changed:
- src/Code.js

What was changed:
- saveListing was refactored to acquire a script lock before selecting the target row.
- clearSalespersonListing was refactored to use the same locked write path.
- updateSalespersonListing was refactored to use the same locked write path.
- Added helper functions:
  - withSheetWriteLock_()
  - getSalespersonListingSheet_()
  - findFirstEmptyListingRow_()
  - writeListingRow_()
- Listing rows are now written as one grouped row write instead of several separate cell writes.

Why this change was made:
- The original code wrote listing data cell by cell.
- That made the save/update/clear flow vulnerable to race conditions and partial writes.
- If two users saved at the same time, they could target the same blank row.
- If execution failed mid-save, one part of a listing could write while another part did not.

What this fixes or improves:
- Reduces the risk of concurrent row collisions
- Reduces the risk of partial row writes
- Makes the listing write path more consistent
- Improves backend reliability under multi-user usage
- Slightly improves performance by reducing repeated write calls

What was preserved:
- Same salesperson tab lookup behavior
- Same columns and same sheet structure
- Same payload fields from the frontend
- Same frontend function names and user flow

Operational result:
- Listing save, update, and clear are now safer and cleaner without changing how users interact with the app.

Change 2: Defensive OpenAI request handling
Date implemented: 2026-04-17
Files changed:
- src/Code.js

What was changed:
- generateListing now delegates the HTTP request/parse logic to a dedicated helper.
- Added constants:
  - OPENAI_API_URL
  - OPENAI_MODEL
  - OPENAI_MAX_ATTEMPTS
- Added helper functions:
  - requestGeneratedListing_()
  - parseGeneratedListingResponse_()
  - parseJsonResponse_()
  - extractGeneratedText_()
  - shouldRetryOpenAiRequest_()

Why this change was made:
- The original OpenAI path assumed the response body could always be parsed as JSON.
- It did not validate HTTP status before treating the response like a success.
- It did not retry transient failures.
- It did not clearly separate request logic from prompt-building logic.

What this fixes or improves:
- Handles malformed JSON responses more safely
- Handles HTTP error codes more explicitly
- Retries temporary 429 and 5xx failures once
- Produces clearer operational errors during API failures
- Makes the code easier to maintain and split later if needed

What was preserved:
- Same generateListing entry point
- Same prompt-building behavior
- Same plain-text listing output goal
- Same frontend usage of the Apps Script function

Operational result:
- AI listing generation is now more resilient and easier to troubleshoot when the external API fails or returns bad data.

Change 6: Execution-scoped backend read optimization
Date implemented: 2026-04-17
Files changed:
- src/Code.js
- tests/spreadsheet.behavior.test.js
- tests/support/mock-spreadsheet.js

What was changed:
- Added EXECUTION_CACHE to reuse spreadsheet and sheet lookups during one Apps Script execution.
- Added getSheet_() so common sheet resolution is cached by region and sheet name.
- Added getSalespersonConfig_() so the Salespeople sheet is read once and reused for both dropdown data and salesperson tab lookup.
- Updated getRegionSpreadsheet_() to cache SpreadsheetApp.openById results per region.
- Updated getSalespeople_(), getSalespersonTabName_(), getSalespersonListings(), getAnalyticsData(), getDataSheet_(), getSalespeopleSheet_(), getHelperSheet_(), and getManagerViewData() to use the new internal caching path.
- Updated getBoatDetails() so it checks the stock-number column first, then reads the matching detail row instead of scanning every full detail row.
- Added local tests proving spreadsheet open and Salespeople sheet resolution are reused within one execution.

Why this change was made:
- The app was doing repeated spreadsheet opens and repeated sheet lookups inside the same request path.
- The Salespeople sheet was being reread and rescanned for related operations that could share the same parsed config.
- Boat detail lookup was reading and iterating the full UIMT detail block when a stock-number-first lookup could reduce work safely.

What this fixes or improves:
- Reduces redundant backend work during region load and related selector flows
- Makes salesperson lookup and tab resolution more efficient
- Makes boat detail lookup lighter for large UIMT sheets
- Improves perceived responsiveness without changing what data is returned
- Keeps the performance gains inside backend internals instead of changing the UI contract

What was preserved:
- Same Google Sheets source spreadsheets and sheet names
- Same response shapes returned to the frontend
- Same selector behavior and frontend flow
- Same Google Sheets read semantics from the user's perspective

Operational result:
- The backend should feel smoother during selector-driven workflows while preserving the same user-facing behavior.

Change 7: Salesperson dropdown load optimization
Date implemented: 2026-04-17
Files changed:
- src/Code.js
- src/Index.html
- src/Listings.html
- tests/spreadsheet.behavior.test.js

What was changed:
- Added lightweight server wrappers:
  - getRegionSalespeople(region)
  - getRegionClasses(region)
- Updated Index.html so salespeople and classes load in parallel instead of one combined request delaying both.
- Updated Listings.html so region change loads only salespeople and no longer waits on unused class-loading work.
- Added stale-response protection using request tokens so quick region changes do not let old responses overwrite the current dropdown state.
- Added a runtime test covering the new lightweight selector endpoints.

Why this change was made:
- The salesperson dropdown was still feeling slow after region selection.
- The main reason was that salespeople were waiting for class loading to finish in the same request cycle.
- On the Listings page, the app was still requesting classes even though the page never uses them.

What this fixes or improves:
- Salespeople can populate sooner after region selection
- Region-change responsiveness is better on both the Listing Generator and Listings page
- The Listings page does less unnecessary backend work
- Rapid region switching is safer because stale responses are ignored

What was preserved:
- Same salesperson names
- Same class data
- Same Google Sheets sources
- Same UI controls and same user workflow after the dropdowns load
- Same backend logic for how salespeople and classes are derived

Operational result:
- The salesperson dropdown should feel faster because it is no longer blocked by class-loading work.
